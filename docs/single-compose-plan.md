# Plan — "Just a docker-compose file" deployment

## Goal

Make SUB/WAVE deployable from a single `docker-compose.prod.yml` + a single `.env`,
both sitting at the repo root. No clone required for prod; an operator can:

```bash
mkdir subwave && cd subwave
curl -O https://raw.githubusercontent.com/perminder-klair/subwave/main/docker-compose.prod.yml
curl -O https://raw.githubusercontent.com/perminder-klair/subwave/main/.env.example
mv .env.example .env   # fill in NAVIDROME_*, ADMIN_*, SITE_URL
docker compose -f docker-compose.prod.yml up -d
```

…and have a working radio station.

Dev gets the same `.env`-at-root treatment so there's one config surface across
all three composes (dev, prod, byo-proxy). Dev still uses bind mounts for
hot-reload of `radio.liq` / `Caddyfile`; only prod bakes everything into images.

## Decisions locked in

| Question | Choice |
|---|---|
| Config surface | Single `.env` at repo root (replaces `docker/.env` + `controller/.env`) |
| Icecast passwords | Auto-generated by the new icecast image on first boot, written to `state/icecast-secrets.env`, sourced by liquidsoap + controller entrypoints |
| Scope | Both `docker-compose.prod.yml` and `docker-compose.byo-proxy.yml` |
| Dev mode | Also unified onto root `.env`; bind mounts preserved for hot-reload |

## Current dependencies the prod compose has on the repo

`docker/docker-compose.prod.yml` references all of these and breaks without them:

| Dependency | Today | After |
|---|---|---|
| `./Caddyfile` | Bind-mounted from `docker/Caddyfile` | Baked into `ghcr.io/perminder-klair/subwave-caddy` |
| `../liquidsoap/radio.liq` | Bind-mounted from repo | Baked into `subwave-liquidsoap` image. Dev still bind-mounts it. |
| `../sounds/` | Bind-mounted into both liquidsoap + controller | `COPY`'d into both images. Dev still bind-mounts. |
| `${STATE_DIR}/icecast.xml` | Rendered by `scripts/setup.sh` from a template before first boot | New `subwave-icecast` image renders it on boot via `envsubst` entrypoint |
| `../controller/.env` | `env_file:` directive | Replaced by `environment:` block reading from root `.env` |

## Icecast secrets — coordination details

The icecast image is the canonical source of `ICECAST_*_PASSWORD`. Flow on first boot:

1. `docker compose up -d` starts services in dependency order:
   - `icecast` (no deps) → starts first.
2. Icecast entrypoint:
   ```sh
   SECRETS=/var/sub-wave/icecast-secrets.env
   if [ ! -f "$SECRETS" ]; then
     umask 077
     {
       printf 'ICECAST_SOURCE_PASSWORD=%s\n' "$(openssl rand -hex 16)"
       printf 'ICECAST_ADMIN_PASSWORD=%s\n'  "$(openssl rand -hex 16)"
       printf 'ICECAST_RELAY_PASSWORD=%s\n'  "$(openssl rand -hex 16)"
     } > "$SECRETS"
   fi
   . "$SECRETS"
   envsubst < /etc/icecast2/icecast.xml.template > /etc/icecast2/icecast.xml
   exec icecast2 -c /etc/icecast2/icecast.xml
   ```
3. `liquidsoap` waits on `depends_on: [icecast]`. Its entrypoint blocks until `/var/sub-wave/icecast-secrets.env` exists, sources it, then execs liquidsoap. (Short bounded wait — 30s, then bail loudly.)
4. `controller` waits on `depends_on: [liquidsoap]`; its entrypoint does the same source + exec.

**Operator override:** if the root `.env` declares `ICECAST_SOURCE_PASSWORD` etc., compose's `environment:` block passes them into the icecast container. The icecast entrypoint then **uses the env-provided values** instead of generating new ones. This means a user who really wants to set them in `.env` can; the magic only kicks in when they're blank.

**Reset path:** to rotate Icecast passwords, delete `state/icecast-secrets.env` and restart the stack. Documented in CLAUDE.md.

## Migration for existing operators

Anyone already running with `docker/.env` + `controller/.env`:

1. `scripts/setup.sh` is updated to detect the old layout and merge both into a root `.env` (idempotent, won't overwrite if root `.env` already exists).
2. The old `docker/.env` and `controller/.env` files are NOT auto-deleted — the migration is one-way, but the old files become unused. Document this in the release notes.
3. `state/icecast.xml` becomes unused but is harmless; can be deleted manually.
4. CLAUDE.md gets a migration note.

## Tradeoffs accepted

1. **Editing `radio.liq` in prod now requires an image rebuild**, not a `docker compose restart liquidsoap`. Dev compose keeps the bind mount, so day-to-day Liquidsoap work is unaffected. Acceptable because prod tracks tagged releases (`SUBWAVE_VERSION`) anyway.
2. **Editing `Caddyfile` in prod requires an image rebuild.** Same reasoning. Caddy config changes are rare.
3. **5 images to publish instead of 3.** Caddy + Icecast images are tiny and rarely change; cache will hit on most builds.
4. **`docker compose` invocation moves to repo root**, breaking muscle memory for `cd docker && docker compose ...`. README + CLAUDE.md updated; no functional impact.
5. **`web/.env.local` stays separate** because Next.js native `npm run dev` reads from `web/` and we don't want to teach it about a root `.env`. Documented.

## Order of work (when implementing)

1. Add new Dockerfiles + entrypoints (caddy, icecast). Confirm they build and the icecast image generates + applies secrets on first boot.
2. Update `Dockerfile.liquidsoap` + `Dockerfile.controller` to bake `radio.liq` / `sounds/` + add secrets-source entrypoint.
3. Write the three new compose files at repo root with the unified `environment:` blocks.
4. Verify the prod stack boots from scratch with an empty `state/` and an `.env` that has only Navidrome + admin creds set (Icecast passwords blank).
5. Verify dev compose still hot-reloads `radio.liq` edits.
6. Update `setup.sh` to generate root `.env` only.
7. Update `update.sh`, `generate-jingles.sh`, `health-check.sh` for the new compose paths.
8. Update `.github/workflows/publish-images.yml` matrix.
9. Update CLAUDE.md + README.md.
10. Delete the now-empty `docker/docker-compose*.yml` files (after the moved ones are confirmed working).

## Out of scope

- Multi-arch (arm64) images — Piper/Kokoro/Chatterbox wheels are amd64-only; same constraint as today.
- A `curl | sh` installer — easy to add later once the two-file install works.
- Renaming any of the three first-party images.
