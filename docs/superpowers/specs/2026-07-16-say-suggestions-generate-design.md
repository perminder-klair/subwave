# Generated manual-voice suggestions — design

**Date:** 2026-07-16
**Status:** approved direction (refresh button + server cache), spec under review

## Problem

The "Manual voice DJ" card on `/admin` (dash) shows six canned prompt chips
(`SAY_SUGGESTIONS` in `web/components/admin/DashPanel.tsx`). They set the tone
well, but they never change: after a few visits the operator has seen them all,
and none of them react to what the station is actually doing (weather, time of
day, the active show, what's playing).

## Goal

Fresh suggestion chips in the same house voice, generated by the station's
configured LLM on demand, with the current six as the style anchor and the
permanent fallback. No LLM call, no latency, and no token spend on ordinary
page opens.

## Decision (made with operator)

**Refresh button + server cache.** The chip row renders instantly from the
last generated batch (server-cached in memory; the hardcoded set until the
first generation or after a controller restart). A small ↻ button on the row
generates a fresh batch on demand. Generation is an explicit operator action,
so — like the manual `/dj/segment` runners — it is **not** budget-gated.

Rejected alternatives: *generate on every page open* (one LLM call per visit,
chips pop in late on slow local models, needs budget gating) and
*stale-while-revalidate* (auto background calls; more moving parts and the
same budget question, for marginal freshness gain).

## Design

### Controller — prompt layer

New generator in `controller/src/llm/internal/prompts/generate.ts`, exported
through the `llm/dj.ts` barrel alongside `generatePersona`/`generateShow`/
`generateTheme` (same `djObject` + Zod structured-output path):

```
generateSaySuggestions(): Promise<string[]>   // kind: 'generateSaySuggestions'
```

- **Schema:** six NAMED string slots (`s1`–`s6`, each `.catch('')`), not an
  array — verified during implementation: given an array field, a 4B local
  model reliably emitted one item and stopped; six required fields make it
  fill six. Same "singular is kinder to weak local models" reasoning as
  `SHOW_SCHEMA`.
- **Prompt inputs:**
  - The six canonical example lines (moved to this module as
    `SAY_SUGGESTION_EXAMPLES` — the controller-side canonical copy) as few-shot
    style anchors, with an instruction *not* to repeat them verbatim.
  - Station context from `context.getFullContext()` — time of day, weather,
    festival, dominant mood.
  - Now playing (title/artist from queue state) and the active show name +
    on-air persona name when a show is on, so chips can be topical.
- **Instruction shape:** write 6 one-line *directions to the DJ* (not verbatim
  copy): imperative, playful, ≤ 90 characters, no emoji, no numbered list —
  they are fed to the styled manual-say path, which rewrites them in persona.

### Controller — route + cache

`POST /generate/say-suggestions` in `controller/src/routes/generate.ts`
(admin-gated like its siblings; no body):

- Calls `dj.generateSaySuggestions()` — up to twice, MERGING the batches: a
  small local model routinely under-delivers, and two half-batches at
  temperature 0.9 usually add up. Bounded at 2 calls (explicit operator
  click; the configured model may be slow).
- **Soft-normalises** the merged batch (same spirit as `generate/show`): trim,
  strip wrapping quotes/bullets, drop empties, dupes, and lines > 120 chars.
  If fewer than 3 survive → treat as failure (500), keep the old cache.
- On success stores the batch in a **module-level in-memory cache**
  `{ suggestions, generatedAt }` and returns
  `{ ok: true, suggestions, generatedAt }`.

`GET /generate/say-suggestions` (admin-gated): returns the cached batch
(`{ ok: true, suggestions, generatedAt }`) or
`{ ok: true, suggestions: null }` when nothing has been generated since boot.
No LLM call ever happens on GET.

Cache is deliberately **not persisted** to `state/` — after a controller
restart the dash simply shows the hardcoded set until the operator hits ↻
again. (YAGNI; persistence can be added later if restarts prove annoying.)

### Web — DashPanel

- `saySuggestions` state initialised to the existing `SAY_SUGGESTIONS`
  constant (kept in the component as the zero-latency initial render and the
  offline/error fallback; a comment cross-references the controller-side
  canonical copy).
- On mount, one `GET /generate/say-suggestions` alongside the existing dash
  fetches; if it returns a batch, replace the chips. Failure or `null` → keep
  the hardcoded set silently.
- A small ↻ icon button (`RefreshCw`, spinning while busy) at the end of the
  chip row fires `POST /generate/say-suggestions`; on success the chips swap
  in place. On failure: error toast (same sonner pattern as the rest of the
  dash), chips unchanged. Button disabled while a generation is in flight.
- Chip click behaviour unchanged: fill textarea, flip mode to Styled, never
  send.
- All fetches ride the existing admin client/auth plumbing used by the other
  dash calls.

### Error handling summary

| Failure | Behaviour |
| --- | --- |
| LLM call throws / times out | POST 500s; web toasts; chips unchanged |
| Model returns junk (< 3 usable lines) | Same as above; cache untouched |
| Controller restarted | GET returns null; hardcoded chips until next ↻ |
| Controller unreachable | Mount GET fails silently; hardcoded chips |

### Testing / verification

- `npm run lint` in `controller/` and `web/` (the merge gate; no test runner).
- Manual: dev stack, open `/admin`, confirm hardcoded chips render instantly,
  hit ↻, confirm a fresh on-voice batch lands and a reload serves the cached
  batch without a model call (check `/admin/debug` LLM log — exactly one
  `generateSaySuggestions` entry per ↻ press).

## Out of scope

- Persisting the cache across restarts.
- Auto-regeneration on any schedule or on page open.
- Per-show or per-persona suggestion presets.
